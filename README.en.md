---

Design systems used to be for humans only. Read the style guide, infer a component's intent, judge by context тАФ that was the work of designers and engineers.

But UI is no longer written by humans alone. When an AI generates the code, picks the components, and lays out the page, a design system being **human-readable is no longer enough.**

melta UI's thesis: **add machine-readability without sacrificing human-readability.** Both at once.

---

## Architecture тАФ three layers

```
Layer 1 тАФ Constitution (what the AI reads first)
  DESIGN.md          Brand identity + 7 principles + Quick Reference (enough to generate basic UI)
  CLAUDE.md / AGENTS.md   Working instructions & reading guide (agent-neutral SSOT = AGENTS.md)

Layer 2 тАФ Specification (machine-readable SSOT)
  design/contracts/
    тФЬтФАтФА tokens.json        99 design tokens (+ tokens.dtcg.json тАФ W3C DTCG 2025.10 export)
    тФЬтФАтФА rules.json         99 prohibition rules (id + severity + detector)
    тФФтФАтФА components/        33 contracts (28 web + 5 React-Native-pending): variants + sizes + a11y + rules

Layer 3 тАФ Verification (violations don't pass)
  scripts/design/    validate / drift-check / lint-generated
  tests/             Playwright + axe-core (170 tests)
  .github/workflows/ CI runs all of the above on every push/PR
```

| Layer | Format | Reader | Role |
|-------|--------|--------|------|
| **DESIGN.md** | Markdown | every AI agent | Design constitution + Quick Reference. Enough on its own to generate basic UI |
| **AGENTS.md / CLAUDE.md** | Markdown | AI (Codex/Cursor/Claude) | Working steps, reading modes, npm scripts |
| **contracts/** | JSON | AI + harness | Strict spec: 33 contracts (28 web) + 99 rules + 99 tokens |
| **harness** | TypeScript | CI | Schema validation, drift detection, Playwright + axe |
| **MCP server** | TypeScript | AI agents | Exposes token lookup / component fetch / rule & HTML checks as tools |

## Why it's readable for AI

1. **Progressive loading** тАФ don't waste context.

   | Mode | Files read | Use |
   |------|------------|-----|
   | Quick | `DESIGN.md` only | single component |
   | Standard | + `theme.md` + relevant contracts / component md | a page |
   | MCP | `get_token` / `get_component` / `check_rule` / `check_html` / `get_rules` / `search` | tool integration |
   | Full | everything | bootstrapping a new project |

2. **Machine-readable specs тАФ reference, not interpretation.** Every component is a contract (`*.contract.json`) with variants, sizes, a11y, token refs and the exact Tailwind string to copy.

3. **99 prohibition rules** with explicit `detector` + `alternative`, so an AI that guesses wrong is caught тАФ by CI, by the MCP `check_rule` tool, and by the Claude Code hook.

4. **Self-verification in the loop** тАФ the MCP `check_html` tool runs the *same* lint as CI, so an agent can generate тЖТ check тЖТ fix without leaving the conversation.

## Enforcement тАФ the DS ships its own guards

| Layer | For | How |
|-------|-----|-----|
| **PostToolUse hook** | Claude Code | Shipped in `.claude/settings.json`. On Write/Edit of HTML, lint runs: `error` blocks (the agent auto-fixes), `warn` is injected as advice |
| **CI** | every agent | `.github/workflows/design-check.yml` checks changed files for prohibited patterns on each PR/push |
| **CLI** | Codex / Cursor / etc. | `npm run design:lint-generated -- <file>` тАФ wire into any agent's hook system |

### Verification coverage тАФ a route-based KPI (not a single number)

Auto-generated by `npm run design:coverage` and CI-guarded for freshness, so the numbers move every time the harness improves rather than being a static claim.

<!-- BEGIN:coverage-en (npm run design:coverage уБзхЖНчФЯцИР) -->
| Route | Count | What |
|-------|-------|------|
| Static auto-detection | **41 / 99** | class-match 31 (same path as MCP `check_rule`) + html-attr 5 + composition 5 (nesting + a11y DOM) |
| Interaction test | 1 | `tests/modal.spec.ts` verifies focus trap / Escape / focus return in a real browser |
| Statically undetectable | 3 | `impossible-static` (active/selected/current are semantically dependent) |
| Manual (AI reference only) | 54 | Context-dependent; surfaced to the AI via `get_rules` |
<!-- END:coverage-en -->

The same orphan-0 drift check guarantees every "manual" rule stays reachable from a contract or doc тАФ so no rule silently becomes dead as the registry grows.

## Distribution

- **npm** тАФ `npx -y melta-ds-mcp` runs the MCP server standalone; `melta-contracts` ships the tokens/rules/contracts JSON (framework-agnostic, consumed by web *and* React Native).
- **MCP Registry** тАФ `io.github.tsubotax/melta-ui`.
- **Web** тАФ `DESIGN.md`, `AGENTS.md`, `llms.txt` / `llms-full.txt` served at https://melta.tsubotax.com.

### Multi-platform contracts

One contract package feeds two implementations: web (this repo, HTML + Tailwind) and [melta-app](https://github.com/tsubotax/melta-app) (React Native). Contracts are split into a **normative core** (variant vocabulary, states, tokenRefs, a11y тАФ shared across platforms; `platformSemantics` declares where divergence is legitimate, e.g. hoverтЖТpressed, elevation decomposition, 44pt touch targets) and **platform recipes** (`recipes/web/` generated from contracts and freshness-checked in CI; `recipes/app/` hand-authored React Native styleRefs, 100% token references). Enforcement runs both ways: `design:compat` diffs the published npm package against HEAD and forces semver bumps on breaking changes, while melta-app's consumer tests go red if web breaks a token, a contract key, or version sync.

## Quick Start

**Claude Code** тАФ drop the repo in your project root; Claude auto-loads `DESIGN.md` + `CLAUDE.md`. Just describe the UI.

**MCP (any agent)** тАФ add the server:
```jsonc
{ "mcpServers": { "melta-ui": { "command": "npx", "args": ["-y", "melta-ds-mcp"] } } }
```

**Contracts only** тАФ `npm install melta-contracts` and read `tokens.json` / `rules.json` / `components/*.contract.json` directly.
